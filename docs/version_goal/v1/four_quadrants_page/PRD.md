# 四象限功能产品需求文档 (PRD)

## 1. 文档概述

### 1.1 背景
高效的时间管理是提升个人生产力的关键。时间管理领域著名的“艾森豪威尔矩阵”（Eisenhower Matrix，又称四象限法则）将任务按照**重要性**和**紧急性**两个维度进行划分，帮助用户快速识别核心任务，合理分配精力。为了帮助用户更科学地管理待办事项，本项目决定在现有任务管理系统的基础上，新增“四象限”视图功能。

### 1.2 目的
本文档旨在明确“四象限”页面的功能需求、交互设计及数据模型定义，为后续的研发、测试提供参考依据。

---

## 2. 用户角色与使用场景

| 角色 | 典型场景 | 核心诉求 |
| :--- | :--- | :--- |
| **职场人士 / 学生** | 面对海量待办事项无从下手，容易陷入“救火”状态（只做紧急的事）或拖延重要非紧急的事。 | 需要直观的看板将任务按优先级归类，一目了然地识别出每天最核心的“重要且紧急”（高优先级）任务，并能方便地进行跨象限调整。 |
| **重度时间管理爱好者** | 有自己的一套时间看板管理逻辑，习惯每天早晨规划四象限任务。 | 希望能够直观地在四象限中拖拽管理不同优先级的任务，并能快速新建任务到指定象限。 |

---

## 3. 产品定位与页面结构

基于 `default.jpg` 和 `more_features.jpg` 的设计，四象限页面作为独立视图或独立路由页面引入，采用经典的 **2x2 网格布局**，四个象限与任务的优先级（TaskPriority 枚举）一一对应：
1. **第一象限（重要且紧急）**：对应 **高 (High)** 优先级。位于左上角，采用红色系主色调。
2. **第二象限（重要不紧急）**：对应 **中 (Medium)** 优先级。位于右上角，采用黄色/橙色系主色调。
3. **第三象限（紧急不重要）**：对应 **低 (Low)** 优先级。位于左下角，采用蓝色系主色调。
4. **第四象限（不重要不紧急）**：对应 **无 (None)** 优先级。位于右下角，采用绿色/灰色系主色调。

---

## 4. 功能需求详情

### 4.1 四象限看板视图（主界面需求）
根据 `default.jpg`，主界面需要满足以下需求：

#### 4.1.1 页面布局与适配
*   **2x2 网格结构**：屏幕主体区域等分为四个象限，在平板或大屏上固定为 2x2 网格；在移动端竖屏下，每个象限高度应自适应，整体填满屏幕可用区域。
*   **象限卡片**：每个象限为一个卡片容器，包含：
    *   **象限头部**：显示象限名称（如“重要且紧急”）、当前象限内**未完成任务数**，以及该象限专属的背景色/边框色。
    *   **快速添加按钮（+）**：点击可在该象限内快速弹窗/拉起输入框创建任务，新任务默认获得该象限对应的优先级。
    *   **任务列表**：展示该象限内的任务，支持纵向滚动。

#### 4.1.2 任务卡片展示样式
*   **任务项包含**：
    *   复选框：支持点击直接勾选完成/取消完成，完成时带有划线效果。
    *   任务标题：文本展示，超出长度限制时省略号截断。
    *   辅助信息图标：如截止时间（过期显示红色）、标签、子任务进度等。
*   **列表排序**：默认按任务的自定义顺序排列，支持用户在设置中切换为按“截止时间”、“创建时间”或“手动拖拽”排序。

#### 4.1.3 拖拽交互 (Drag & Drop)
*   **象限内排序**：长按任务卡片，可在当前象限内上下拖动调整顺序（自定义排序模式下生效）。
*   **跨象限移动**：长按任务卡片并拖拽至其他三个象限，释放后**任务的优先级（Priority）属性自动更新为目标象限对应的优先级值**，并实时重绘视图。

---

### 4.2 任务归类规则
四象限的数据分流完全依赖于任务的 **`Priority`（优先级）** 属性，不涉及日期或其他字段的复合判断，不使用独立的 `quadrantIndex` 属性。

| 象限位置 | 象限默认名称 | 对应任务优先级 (TaskPriority 枚举值) |
| :--- | :--- | :--- |
| **左上 (第一象限)** | 重要且紧急 | **High (高)** |
| **右上 (第二象限)** | 重要不紧急 | **Medium (中)** |
| **左下 (第三象限)** | 紧急不重要 | **Low (低)** |
| **右下 (第四象限)** | 不重要不紧急 | **None (无)** |

*注：用户在四象限主界面拖拽任务跨象限移动时，系统会反向修改任务的 `Priority` 属性以符合目标象限。*

---

### 4.3 更多功能与自定义设置
根据 `more_features.jpg`，右上角更多/设置弹窗包含以下高级功能：

#### 4.3.1 视图过滤与筛选
*   **清单过滤**：支持选择仅展示特定清单下的任务，或排除某些特定清单（如排除“垃圾箱”、“已归档”）。
*   **标签过滤**：支持通过标签筛选四象限任务。
*   **隐藏/显示已完成任务**：开关项。开启时，已完成任务在象限中消失；关闭时，已完成任务在象限底部置灰显示。

#### 4.3.2 界面与命名自定义
*   **象限重命名**：用户可以点击修改四个象限的标题名称。例如将“第一象限”重命名为“核心工作”，将“第四象限”重命名为“备忘闲杂”。
*   **象限主题色自定义**：支持为四个象限分别分配预设的颜色主题（红、橙、蓝、绿、灰、紫等）。

#### 4.3.3 数据统计与历史分析
*   **优先级分布统计**：展示不同优先级任务在系统中的分布比例（饼图或百分比条）。
*   **时间精力评估**：如果高优先级任务占比过高（>50%），给出警示提示（“高优先级任务占比过大，可能需要梳理并分流至中/低优先级”）；如果中优先级任务能维持高完成度，则给出正向激励。

## 6. 技术实现要点与数据模型建议

### 6.1 数据模型重构 (Isar Database Entity)
为了支持该模式，我们需要对现有 `Task` 的 `priority` 字段进行重构。

#### 6.1.1 优先级枚举定义 `TaskPriority`
在合适的数据模型层定义任务优先级的枚举类型，使其能被 Isar 识别：
```dart
import 'package:isar_community/isar.dart';

enum TaskPriority {
  none,   // 对应第四象限 (无)
  low,    // 对应第三象限 (低)
  medium, // 对应第二象限 (中)
  high,   // 对应第一象限 (高)
}
```

#### 6.1.2 任务实体 `Task` 修改
将 `Task` 实体中的 `int priority` 修改为 `TaskPriority priority`，并使用 Isar 的枚举注释（通常按 Ordinal 或 Name 存储）：
```dart
  /// 优先级定义
  @Index()
  @enumerated
  TaskPriority priority;
```
同时需要更新 `Task` 构造函数默认值、`copyWith` 方法、`toJson` 和 `fromJson` 方法中的序列化/反序列化逻辑，以适配 `TaskPriority` 枚举类型。

#### 6.1.3 象限配置实体 `QuadrantConfig` (新增)
用于持久化用户的自定义象限样式和过滤配置：
```dart
@collection
class QuadrantConfig {
  Id id = Isar.autoIncrement;

  // 四个象限的自定义名称，默认为 ["重要且紧急", "重要不紧急", "紧急不重要", "不重要不紧急"]
  List<String> names = ["重要且紧急", "重要不紧急", "紧急不重要", "不重要不紧急"];

  // 四个象限的主题色十六进制值或色值索引
  List<int> themeColors = [0xFFE57373, 0xFFFFB74D, 0xFF64B5F6, 0xFF81C784];

  // 是否隐藏已完成任务
  bool hideCompleted = true;

  // 过滤的清单 ID 列表（若为空则展示全部）
  List<int> filteredListIds = [];
}
```

### 6.2 业务逻辑联动 (`TaskProvider` 状态管理)
*   **四象限过滤与投递**：
    *   `TaskProvider` 从 Isar 获取未归档的任务数据。
    *   提供 Getter 方法，根据 `QuadrantConfig` 配置条件，把任务按照 `priority` 映射投递：
        *   `TaskPriority.high` $\rightarrow$ 第一象限列表
        *   `TaskPriority.medium` $\rightarrow$ 第二象限列表
        *   `TaskPriority.low` $\rightarrow$ 第三象限列表
        *   `TaskPriority.none` $\rightarrow$ 第四象限列表
*   **跨象限拖拽的物理更新**：
    *   在主界面将任务由象限 A 拖拽到象限 B 时，UI 触发更新，通过 `TaskProvider` 或 `TaskLifecycleManager` 修改目标任务的 `priority` 属性为象限 B 对应的 `TaskPriority` 值。
    *   调用数据库保存并触发 UI 通知，所有依赖视图（如四象限看板、普通任务列表、看板视图）自动实现同步刷新。

## 7. 测试要点 (QA Focus)
1.  **优先级枚举正确性**：修改 `priority` 为枚举后，验证原有任务的创建、编辑、详情显示、列表分组、排序功能是否依然正常。
2.  **四象限分流一致性**：创建不同优先级的任务，验证它们是否 100% 准确落入对应的象限中，不包含任何日期维度的干扰。
3.  **拖拽优先级变更**：跨象限拖动任务，验证其在数据库中保存的 `priority` 值是否发生了对应变更，且其他视图（如任务列表）是否同步变化。
4.  **适配性测试**：在不同分辨率和尺寸的手机设备上测试 2x2 网格容器在任务较多时的滚动流畅度与界面防爆体验（防止溢出）。
