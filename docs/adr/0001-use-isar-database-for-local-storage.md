# ADR 0001: 使用 Isar 数据库进行本地结构化存储

## 状态
已接受 (Accepted)

## 背景
“My Dida” 作为一个效能管理工具，具有较多的关联数据类型：
* `Task` 和 `Checklist` 存在一对多的父子关联关系。
* `Task` 和 `CheckPoint` 存在一对多的内嵌关系。
* `Habit` 和 `HabitCheckInRecord` 存在一对多的日志关联关系。
* 还有番茄钟记录、侧边栏配置等多种表。

在 Flutter 生态中，常用的本地存储选项有：
1. **SQLite (sqflite)**：经典的关系型数据库，功能强大，支持复杂关联，但需要编写繁琐的 SQL 语句进行表迁移和数据映射，对于嵌套结构（如 CheckPoint）维护成本高。
2. **Hive**：轻量级的 NoSQL 键值对数据库，速度极快，但缺少强类型 Schema 约束，且不支持复杂的条件过滤和关系查询，不适合存储复杂关系的数据模型。
3. **Isar 数据库**：专门为 Flutter 设计的超快 NoSQL/对象数据库。

## 决策
我们决策使用 **Isar 数据库 (`isar_community`)** 作为“My Dida”项目的核心本地存储方案。

选择它的主要考量：
* **强类型与代码生成**：通过 `build_runner` 生成类型安全的表结构文件（`*.g.dart`），开发体验良好。
* **内嵌对象支持**：可以直接在 `Task` 实体类中定义 `List<CheckPoint>` 嵌入类（Embedded），Isar 会自动处理嵌套序列化，避免了为简单的子项列表单独建表的麻烦。
* **优秀的关联查询 (Links)**：支持通过 `IsarLink` 和 `IsarLinks` 来定义实体之间的关联（例如 `Checklist` 与 `Task` 的双向链接），支持简洁的链式过滤。
* **异步与事务支持**：Isar 原生支持多线程异步读写操作以及 ACID 事务，保障了数据的强一致性。

## 后果
* **正面影响**：
  - 数据模型层代码简洁，表结构更改时只需运行 `dart run build_runner build` 即可完成 Schema 的更新和迁移。
  - 数据读取效率极高，支持复杂的级联过滤（比如在日历页面进行时间范围和特定清单的复合过滤）。
* **负面影响**：
  - 项目对 `build_runner` 存在强依赖。每次修改实体模型文件时，必须重新生成代码，开发时需要开启 `build_runner watch`。
